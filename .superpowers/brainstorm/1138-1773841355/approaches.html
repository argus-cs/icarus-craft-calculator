Abordagens de Arquitetura

Como estruturar o Icarus Crafting Calculator

A

Monolito Simples

Scraper + App tudo junto no mesmo repo Nuxt

  • Script de scraping como scripts/scrape.ts que gera JSON em public/data/
  • App Nuxt lê os JSONs como assets estáticos
  • Uma única página com lista + busca, detalhes inline e modal de materiais
  • SSG gera site estático no build

✓ Prós

  • Simples de manter
  • Um repo só
  • Deploy direto

✗ Contras

  • Scraper misturado com app
B

Separação Scraper / App

Dois pacotes: scraper independente + app Nuxt

  • Monorepo com packages/scraper e packages/app
  • Scraper gera JSON, app consome
  • Scraper pode rodar via CI/CD para atualizar dados
  • App Nuxt permanece leve e focado na UI

✓ Prós

  • Separação de responsabilidades
  • Scraper reutilizável
  • CI pode automatizar updates

✗ Contras

  • Mais setup inicial
  • Complexidade de monorepo
C

Monolito com Scraper como Módulo Nuxt

App Nuxt + scraper como server route / módulo interno

  • Scraper como server/scripts/scrape.ts ou módulo Nuxt
  • Pode expor endpoint /api/scrape em dev para atualizar
  • Dados gerados em public/data/
  • Em produção funciona como site estático normal

✓ Prós

  • Integrado mas organizado
  • Fácil de rodar scraper

✗ Contras

  • Acoplamento desnecessário
  • Complexidade sem ganho real

⭐ Recomendação: Opção A

Para o escopo deste projeto (3 categorias de itens, sem backend, dados estáticos), a simplicidade do monolito é a melhor escolha. O scraper é um script utilitário que roda raramente — não justifica a complexidade de um monorepo. Um scripts/ no mesmo repo resolve perfeitamente.
